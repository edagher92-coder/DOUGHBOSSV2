1The 30-second overview

DoughBoss is one system with three faces. A customer never sees your control room, and you rarely need to touch the parts customers see — but they're all talking to each other, live, the whole time.

CUSTOMERS
Order on the website
Menu, pizza builder, cart, checkout
→
INSTANTLY
Order lands in your control room
wp-admin Orders + the Live Order Board
→
KITCHEN
Staff accept & prepare
One tap: New → Preparing → Ready
→
CUSTOMER
Tracks their order
Order number + email, on the website

Everything in this guide is one of these three faces: the website (what customers use), the control room — WordPress's admin area, called "wp-admin" — (what you, the owner/manager, use), and the Staff Console app (a lighter-weight tool for kitchen/counter tablets that don't need a full login).

2How to log in — the logins you'll use

WhoWhereWhat you log in with
OWNER Owner / ManagerThe control room (wp-admin) at your website address, e.g. https://doughboss.com.au/wp-admin/Your existing WordPress email + password. This is the same login you already use to manage the site — DoughBoss just adds a new menu to it.
STAFF Kitchen / counter staffEither the control room's Live Order Board, or the separate Staff Console app on a tablet/phoneA staff login created for them (see §7) — never your own owner login.
STAFF Till / voucher scanningStaff Console app, or the "Voucher Scan" screen inside wp-adminSame staff login as above — one login covers both the order board and voucher scanning.
What is "wp-admin"? It's WordPress's built-in control panel — the same one you'd use to edit any page on the website, just with a "DoughBoss" section added to the left-hand menu. If you've ever logged in to change something on the site before, you already know how to get here.
What is a Staff Console "Application Password"? It's a special password generated just for one device/app — not your everyday WordPress password. Think of it like a spare key cut specifically for the kitchen tablet: if that tablet is ever lost, you can cancel just that one key without changing anyone else's password. Your admin sets this up once per device (§7).

3Your control room (wp-admin), screen by screen

Once logged in, look for DoughBoss in the left-hand menu. Everything below lives under it.

Orders

What it's for: a list of every order — pickup or delivery, paid or unpaid, with the customer's details and what they ordered.
How to use it: click DoughBoss → Orders. Click any order to see its full detail; change its status (e.g. mark it Ready) directly from the list.

Live Order Board (the Kitchen Board)

What it's for: the big-screen view for the kitchen — new orders appear automatically in a "New" column, and one tap moves them to "Preparing" then "Ready."
How to use it: click DoughBoss → Live Order Board. Leave it open on a kitchen tablet or monitor all shift — it refreshes itself.

Shops (Locations)

What it's for: if you run more than one shop, this is where each one's name, address, phone, and which suburbs it delivers to are set up.
How to use it: click DoughBoss → Shops. Add a shop, or edit an existing one's details and delivery postcodes.

Vouchers

What it's for: managing discount codes — both the ones customers claim themselves online, and one-off codes you create by hand.
How to use it: click DoughBoss → Vouchers. Use "Claim a voucher for a customer" when a real customer calls in or asks in person — this is the option that also sends the code to the till system automatically. The separate "Create a voucher (manual, one-off)" section is for internal/goodwill codes and does not reach the till automatically — use it only when that's not needed.
This distinction trips people up, so it's worth repeating here: only "Claim a voucher for a customer" and the website's own voucher-claim button push a code to the till system. The manual "Create a voucher" tool is for internal use and stays WordPress-only unless someone scans it in-store.

Catering

What it's for: catering enquiries — quotes, deposits, and balance payments for bookings.
How to use it: click DoughBoss → Catering to see enquiries and update their status (e.g. mark a deposit received) as a booking progresses.

Settings

This is the one screen with several tabs — take it slowly, nothing here needs changing often.

TabWhat it controls
StoreCurrency, tax/GST rate, delivery fee, opening/closing ordering, pickup vs delivery toggles, the email address that gets notified of new orders.
Payments (Stripe)Turns on card payments at checkout. Off by default — customers pay on pickup/delivery until this is switched on and the keys from your Stripe account are entered.
POSPal POSConnects the voucher system to your till so claimed vouchers show up there automatically. Off by default; needs your POSPal account details entered once.
Real-time & NotificationsOptional extras: instant push updates to the order board, staff phone alerts on new orders, customer SMS on "order ready," and cloud receipt printing. Each is independently off until you choose to turn it on.
Menu (Sizes / Toppings)The sizes and toppings customers can choose in the pizza builder, plus a one-click "Import standard menu" button if you ever need to reset to the standard board menu.

4What your customers see (the website)

None of this needs your day-to-day attention — it's here so you know what a customer experiences when something comes up in conversation.

  • Menu — browses categories and items, each with a price and description.
  • Pizza builder — pick a size, add toppings, see the price update live as they go.
  • Cart & checkout — review the order, choose pickup or delivery, enter their details, pay (by card if Payments/Stripe is switched on, otherwise pay on pickup/delivery).
  • Voucher box — at checkout, a customer can type in a discount code; the saving is worked out on our side, never just trusted from what the customer's screen shows.
  • Order tracking — a customer looks up their order status later using their order number and the email they used.

5The Staff Console app (kitchen & counter tablets)

This is a separate, lightweight app — it's for a shop tablet that shouldn't need a full WordPress login just to watch the order board or scan a voucher.

Installing it: open the Console's web address on the tablet/phone and add it to the home screen like any app — no app-store download needed.
Logging in: enter the staff Application Password set up for that device (see §7) — not a personal password, and not your owner login.
What staff can do there: watch the live order board, and scan customer vouchers (camera QR scan, or type the code in by hand) to redeem them.

6Who should have which login

Login typeCan doGive this to
Administrator (full wp-admin)Everything — Orders, Vouchers, Settings, Payments, POSPal, adding other staff logins.Owner / manager only. Do not hand this out to kitchen or counter staff.
DoughBoss Kitchen roleLive Order Board + redeeming vouchers. No access to Settings, Payments, or creating vouchers.Kitchen and counter staff who need the board and voucher scanning, nothing more.

Never share your own Administrator login with staff. If a staff member needs board/voucher access, create them their own login (§7) instead of handing out yours — that way you can see who did what, and remove just their access if they leave, without having to change your own password.

7Creating a new staff login, step by step

Use this whenever a new staff member joins, a tablet needs setting up, or you want a throwaway login just to try things out yourself.

Log in to the control room (wp-admin) with your own Administrator login.
In the left-hand menu, click Users → Add New.
Fill in a Username (e.g. kitchen-revesby) and an Email address. For a shared kitchen/counter login, a dedicated address like kitchen@doughboss.com.au works well — it doesn't need to be a real inbox anyone checks daily, but must be a validly-formatted email address WordPress will accept.
Click Show password and either keep the strong password WordPress generates, or set your own. Write it down somewhere safe — WordPress will not show it again.
Under Role, choose DoughBoss Kitchen (not Administrator) for kitchen/counter staff.
Click Add New User. That's it — hand the username + password to the staff member, and they can log in at the same control-room address you use.
Setting up the Staff Console app instead? Create the staff user exactly as above, then log in as that user, go to Users → Profile → Application Passwords, give it a name (e.g. "Kitchen tablet — front counter"), and click Add New Application Password. WordPress shows a one-time password — enter that (with the same username) into the Console app on the tablet. It's a different password from the one used to log into wp-admin directly, on purpose.
Want a throwaway "demo" login to explore the system yourself first? Follow the exact same steps above with a username like demo-owner and an email like demo@doughboss.com.au, and choose the Administrator role only if you want to see everything (including Settings/Payments) — otherwise choose DoughBoss Kitchen to see it exactly as staff would. Delete the user from Users afterwards if it was only for a test drive.

9Keeping it secure

  • Never share your Administrator login — create a separate staff login instead (§7), so access can be removed for one person without affecting anyone else.
  • One login per person or per device where practical — it makes it obvious who did what, and what to cancel if a phone/tablet is lost.
  • Staff only ever need the "DoughBoss Kitchen" role (or a Console Application Password) — never give kitchen or counter staff the Administrator role.
  • Payment keys (Stripe, POSPal) are never shown back on screen once saved — if a field looks blank when you return to Settings, that's normal; it means a value is already saved and safely hidden, not that it was lost.
